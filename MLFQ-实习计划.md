## 多级反馈队列（MLFQ）调度器实习计划

### 一、目标与背景

- **实习目标**：在现有基于 Rust 的教学操作系统内核中，实现一个可配置、可验证的多级反馈队列（MLFQ）调度器，并通过实验对比现有调度算法的性能与行为差异，加深对调度策略设计权衡的理解。
- **当前环境**：  
  - 操作系统/内核：`基于 Rust 的教学 OS 内核（例如 rCore / 自研 Rust OS 内核）`  
  - 当前调度策略：`（例如：简单时间片轮转 RR / 先来先服务 FCFS / 基于静态优先级的调度 等）`  
- **引入 MLFQ 的原因**：  
  - 更好地兼顾**交互型任务的响应时间**和**CPU 密集型任务的吞吐量**；  
  - 在 Rust 内核中通过类型安全的数据结构与并发原语，实现“多跑 CPU 的任务逐渐降级”，“经常阻塞 / I/O 型任务保持或提升优先级”，更贴近现代操作系统调度的工程实践。

---

### 二、需求与设计目标

- **功能性需求**
  - 支持 `N` 级就绪队列（N 预期为 3–5 级，可配置）。
  - 不同级别队列具有不同的时间片长度（高优先级队列时间片短，低优先级队列时间片长）。
  - 进程在运行过程中，根据 CPU 使用情况、阻塞/唤醒等事件，在不同级别队列之间移动。
  - 支持周期性“优先级提升”（priority boost），避免低优先级进程长期饥饿。
- **非功能性需求**
  - 尽量复用现有调度框架，减少对内核其他部分的侵入式修改。
  - 代码结构清晰，方便后续修改调度策略、时间片参数和调度准则。
  - 提供简单的统计接口，用于记录每个进程在各级队列的停留情况、被抢占次数等，支撑实验分析。

---

### 三、总体设计思路

#### 3.1 队列层级与数据结构

- **队列层数**
  - 初步计划采用 `3~5` 级反馈队列，例如：`Q0`（最高优先级）到 `Q3/Q4`（最低优先级）。
- **数据结构设计（Rust 表达）**
  - 为每一级维护一个就绪队列：`ready_queues[level]`。
  - 队列底层实现：倾向使用 Rust 标准或内核自定义容器，例如 `VecDeque<Arc<TaskControlBlock>>` 等，结合 `Arc` / 自研引用计数保证任务在多处引用时的所有权安全。
  - 在任务控制块（例如 `struct TaskControlBlock`）中增加字段：
    - `mlfq_level: usize`：当前所在队列级别；
    - `time_slice_used: u64`：当前时间片已使用的 tick 数；
    - `// 可选统计字段`：如累计运行时间、被抢占次数、队列切换次数等，使用 Rust 的整数类型（如 `u64`）表示。

#### 3.2 时间片与调度优先级

- **时间片策略（示例）**
  - `Q0`：最短时间片，例如 1 个 tick；
  - `Q1`：时间片为 `Q0` 的 2 倍；
  - `Q2`：时间片为 `Q0` 的 4 倍；
  - ……
- **调度顺序**
  - 调度器每次从最高优先级的非空队列开始选择进程（例如 FIFO）。
  - 若当前运行进程用完所在级别的时间片，并且该队列中还有其它就绪进程：
    - 进程优先级降低：`mlfq_level++`（不低于最低级队列）；
    - 将进程插入新级别队列的队尾。
  - 若有更高优先级队列变为非空，则允许其**抢占**当前低优先级进程。

#### 3.3 动态反馈与饥饿避免

- **基于 CPU 使用的反馈**
  - 若进程在时间片用完前主动阻塞（例如等待 I/O）：
    - 认为其更偏向 I/O 型，可以在唤醒时保持当前级别，或适度提升优先级（插入高一级队列或本级队列队头）。
- **优先级提升（priority boost）机制**
  - 引入一个全局周期参数 `boost_interval`（以 tick 为单位）：
    - 每隔 `boost_interval`，对所有进程执行一次“优先级重置”或“逐级提升”操作；
    - 将长期处于低优先级队列的进程迁移到高优先级队列。
  - 目标是防止进程被长期压在最低级队列中引起饥饿。

### 四、具体实现步骤

#### 4.1 前期代码阅读与调研

- **调度相关路径（Rust 内核）**
  - 阅读当前调度器入口（例如 `fn schedule()` / `run_next_task()` 等）实现。
  - 找到任务创建（例如 `Task::new()`、`spawn()`）、退出（`exit()`）、阻塞（`sleep()`）、唤醒（`wakeup()`）等关键路径对应的 Rust 函数。
  - 确定时钟中断处理函数 / `timer_tick` 回调所在模块（通常是中断子系统中的 `timer_handler()`），以便在其中集成时间片统计与抢占逻辑。
- **记录关键接口**
  - 在文档中整理目前与调度相关的关键 Rust 模块和数据结构（如 `task`, `sched`, `trap` 模块等），便于后续查阅与修改。

#### 4.2 扩展数据结构

- 在任务控制块 `TaskControlBlock` 中添加 `mlfq_level: usize`、`time_slice_used: u64` 等字段，并在构造函数（如 `TaskControlBlock::new(...)`）中初始化为：
  - 新任务默认进入最高优先级队列 `Q0`；
  - `time_slice_used` 初始化为 `0`。
- 在调度模块中新建多级就绪队列的数据结构：
  - 初始化 `ready_queues[level]: VecDeque<Arc<TaskControlBlock>>` 等结构；
  - 根据现有内核并发模型，使用 `Mutex` / `SpinLock` / 自研锁对队列进行保护；
  - 提供封装的队列操作函数，例如（Rust 伪代码）：
    - `fn enqueue(level: usize, task: Arc<TaskControlBlock>)`  
    - `fn dequeue(level: usize) -> Option<Arc<TaskControlBlock>>`  
    - `fn pick_next_task() -> Option<Arc<TaskControlBlock>>`：从高到低扫描，返回全局下一个要运行的任务。

#### 4.3 修改调度逻辑

- 在调度入口（例如 `fn schedule()`）中：
  - 用“多级队列 + 从高到底扫描”的方式替换原有单一就绪队列逻辑。
  - 切换任务时：
    - 将被选中任务的 `time_slice_used` 重置为 `0`；
    - 根据其当前 `mlfq_level` 决定适用的时间片长度（常量可以定义在 Rust 模块中，例如 `const TIME_SLICE_LEVELS: [u64; N] = [...]`）。
- 在时钟中断处理函数（例如 `fn timer_handler()`）中：
  - 对当前运行任务的 `time_slice_used` 自增。
  - 当 `time_slice_used` 超过当前级别的时间片：
    - 通过设置某个标志位或直接调用 `yield_now()`，标记该任务需要被抢占；
    - 在下一次返回内核调度时，将其降级并放入对应队列队尾。

#### 4.4 阻塞 / 唤醒时的队列管理

- 在任务阻塞（例如 `sleep()` / 等待 I/O）时：
  - 在持有相应锁的前提下，将其从对应就绪队列移除，并将任务状态设置为阻塞态（如 `TaskState::Blocked`）。
- 在任务唤醒（例如 `wakeup()` / I/O 完成回调）时：
  - 根据设定策略决定其重新入队的位置：
    - 同级队列队头 / 队尾；
    - 或适当提升到更高优先级队列（示例：`mlfq_level = mlfq_level.saturating_sub(1)`）。
- 确保在所有状态变迁路径上，就绪队列和任务状态保持一致，避免出现“任务状态是阻塞但仍在就绪队列里”等不一致问题。

#### 4.5 周期性优先级提升实现

- 维护一个全局 tick 计数器 `TICK_COUNT: AtomicU64` 或等价的全局变量，在 `timer_handler` 中自增。
- 每次时钟中断累加 `tick_count`，当 `tick_count % BOOST_INTERVAL == 0` 时：
  - 在合适的上下文中（持有调度相关锁）遍历所有可调度任务（或遍历各级就绪队列）：
    - 将它们的 `mlfq_level` 提升到更高优先级，或统一重置为最高优先级；
    - 重新插入到对应的高优先级队列中。
- 需要注意与调度器、锁机制的配合，避免在遍历队列时发生并发修改问题；利用 Rust 的类型系统保证不会出现悬空引用和数据竞争。

### 五、测试与验证计划

#### 5.1 功能正确性测试

- **基础测试场景**
  - 只运行少量 CPU 密集型进程，观察它们是否按照预期在队列中逐步降级。
  - 混合运行 CPU 密集型和 I/O 密集型进程，观察：
    - I/O 型进程是否能长期停留在高优先级队列；
    - 交互响应（例如终端输入）的延迟是否较原始调度器有所改善。
- **抢占行为验证**
  - 通过增加日志或统计变量，验证：
    - 时间片耗尽时是否触发降级和重新入队；
    - 高优先级进程就绪后，是否能够抢占低优先级进程。

#### 5.2 性能对比与实验

- 设计一组标准化的负载（例如固定数量的 CPU 循环程序、I/O 密集程序、交互程序）。
- 与原有调度算法对比以下指标（可选）：
  - 平均响应时间（Response Time）
  - 平均周转时间（Turnaround Time）
  - 上下文切换次数
- 将结果整理为表格或图表，并在实验报告中分析：

  - 不同队列数、时间片配置对性能的影响；
  - priority boost 周期长短对饥饿现象和整体性能的影响。

#### 5.3 边界情况与稳定性

- 单进程、极少进程、极多进程等不同规模场景下的稳定性。
- 频繁创建和退出进程时，就绪队列是否保持一致、内存是否正常释放。
- 长时间运行（例如几万次调度）后，检查系统是否出现队列损坏或异常调度行为。

### 七、可能问题与应对思路

- **并发与锁问题（Rust 视角）**
  - 在操作多级就绪队列时必须遵守内核的锁顺序和临界区规则，避免在持有 `Mutex` / `SpinLock` 的同时再次触发可能获取同一把锁的路径；
  - 如果出现死锁或竞态，可通过简化执行路径、增加 `log` / `tracing` 输出以及在关键路径加入断言来定位问题。
- **调试难度**
  - 利用 Rust 的日志生态（如 `log` + 后端实现 / `tracing`），在调度关键路径打印当前调度决策、任务 `mlfq_level`、时间片使用情况；
  - 编写专门的用户态测试程序触发边界场景（频繁阻塞、频繁创建/退出等），并在 QEMU/串口输出中观察 MLFQ 队列行为。
- **参数选择困难**
  - 初期采用简单、易解释的默认参数（例如队列数为 3，时间片呈 1:2:4 递增，`BOOST_INTERVAL` 为若干秒对应的 tick 数）；
  - 再通过实验数据逐步调整，并在文档中记录参数变化及其效果，便于社区同学基于相同 Rust 内核复现实验。
