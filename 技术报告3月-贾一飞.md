# 3月技术报告-贾一飞

## 目标与问题

主要目标：

1. 完善多组织共享测试环境的 Cancel 自动解锁与无感集成（对应 [PR #13](https://github.com/arceos-hypervisor/github-runners/pull/13)）
2. 实现依赖感知的测试目标选择（对应 [PR #9](https://github.com/arceos-hypervisor/axci/pull/9)）
3. 完成 EEVDF 调度算法（per-task）实现，并补充验证与演示材料，形成“算法实现 + 单元测试 + 运行时观测”的完整链路（代码分支：[feat/eevdf-class-scheduler](https://github.com/yoinspiration/StarryOS/tree/feat/eevdf-class-scheduler)）。

主要问题：

1. 多组织共享同一块开发板时，虽然已有文件锁串行机制，但当持锁的 workflow 被手动 Cancel 后可能遗留“幽灵锁”，导致后续任务长期卡在 `Waiting for lock`。
2. 测试入口经过模块化重构后，仍缺少稳定的“按变更自动选择目标”能力；很多小改动仍会触发全量测试，造成 CI 资源和开发板占用偏高。
3. EEVDF 机制本身涉及 eligible、deadline 抢占、nice 权重映射等概念，若缺少系统化测试与对照实验，不易直观说明其效果与价值。

## 实现情况

### 多组织共享测试环境（Cancel 自动解锁与无感集成）

- **功能概述**  
  - 在同一台物理主机上，通过 Docker 运行多个 GitHub self-hosted runner 容器，分别注册到不同组织/仓库。
  - 当多个组织需要共享同一块开发板时，使用 `runner-wrapper` + Linux `flock` 文件锁，在 **Job 级别** 串行访问硬件，避免串口、电源等独占资源发生冲突；并新增 watcher 覆盖 Cancel 残留锁场景。
- **整体架构**（对应 `docs/runner-wrapper-multi-org-lock.md`）  
  - 每个组织有自己的一套 runner 实例与 `.env` 配置；runner 容器内部启动时并不直接调用 `run.sh`，而是通过 `runner-wrapper.sh` 注入 Job 前后钩子：
    - `pre-job-lock.sh`：Job 开始前获取锁文件（阻塞等待），启动后台 holder 进程持有锁；
    - `post-job-lock.sh`：Job 结束后写入释放标记，holder 进程退出，flock 自动释放锁。
  - 所有共享同一块板子的 runner，通过**相同的锁 ID** 和 **共享的锁目录**（默认 `/tmp/github-runner-locks`）来协调执行：
    - 例如：`board-phytiumpi.lock`、`board-roc-rk3568-pc.lock`，不同板子使用不同锁文件，可并行。
- **多组织共享的具体用法**（对应 `README_CN.md` 的“多组织共享”小节）  
  - 使用 `runner.sh` 生成环境时，板子 runner 默认启用 wrapper 与锁目录挂载：
    - 通过板子级环境变量（如 `RUNNER_RESOURCE_ID_PHYTIUMPI`、`RUNNER_RESOURCE_ID_ROC_RK3568_PC`）控制锁 ID。
  - 若要在多组织中共享同一块板，只需要在各自组织的 `.env` 中为该板设置**相同的锁 ID**，例如：
    - 组织 A：`RUNNER_RESOURCE_ID_PHYTIUMPI=board-phytiumpi`
    - 组织 B：`RUNNER_RESOURCE_ID_PHYTIUMPI=board-phytiumpi`
  - 这样，即使两个组织的 runner 容器同时收到访问该板的 Job，也会在锁文件上排队，保证同一时刻只有一个 Job 占用硬件。
- **实现特点与限制**  
  - 不额外降低吞吐：串行来自“硬件只能一次服务一个 Job”的物理约束，方案只是把“无序抢占”变成“有序排队”。
  - 不需要额外中间件：锁基于本地文件系统 `flock`，只依赖 Bash + util-linux，部署简单，可以在现有 CI 环境中直接使用。
  - 不涉及改动 GitHub Actions 的调度逻辑：GitHub 仍按组织维度调度 Job，本仓库通过 runner 容器和 job 钩子在主机侧完成多组织共享与冲突避免。
- **Cancel 自动解锁与运维增强**  
  - 新增 `lock-watcher` 机制：从 `${RESOURCE_ID}.holder` 读取持锁 `run_id`，查询 GitHub Actions Run 状态；检测到 `completed + cancelled` 后主动清理 `.holder` / `.release`，避免“幽灵锁”长期占用。  
  - 在 `.env` 配置 `RUNNER_LOCK_MONITOR_TOKEN` 后，`./runner.sh compose` 自动生成 watcher 服务，`./runner.sh start/stop/restart` 无参时与 runner 一起启停，实现“无感使用”。  
  - watcher 支持通过 `RUNNER_RESOURCE_IDS` 单进程监控多个板卡资源；文档与部署建议从“每板一个进程”收敛为“每组织一个 watcher”。  
  - 补充锁目录权限排障与脚本鲁棒性：锁目录不可写时快速失败并给出修复提示；等待锁时输出周期日志，便于定位排队状态。

### 依赖感知的测试目标选择（规则驱动 + Rust 引擎回退）

- **合并与重构背景**  
  - 将主线分支模块化改造（`tests.sh` 拆分为 `lib/*.sh`、CLI 增强、CI 现代化）合入工作分支，并在其上增量移植自动目标选择能力，避免长期维护两套实现。
- **规则驱动自动目标选择**  
  - `tests.sh` 新增 `--auto-target` 与 `--base-ref`，支持基于变更范围自动选择测试目标。  
  - 规则由 `configs/test-target-rules.json` 外置维护，支持路径匹配与 crate 级规则（如 `run_all_crates`、`crate_rules`、`crate_path_rules`）。  
  - 优先使用 `axci-affected` Rust 引擎进行分析，异常时回退到 shell 规则匹配，保证流程可用性。  
  - 目标选择结果写入 `FILTER_SUITE`，与现有 `resolve_suites()` 管线直接对接。
- **CI 集成与稳定性增强**  
  - `test.yml` 新增 `test_targets=auto`、`base_ref`、`runs_on` 等输入，并新增 `detect-targets` job 在 CI 中执行 `--auto-target --list-auto`。  
  - CI 输出 `selection_mode`、`auto_reason` 与目标列表到 `GITHUB_STEP_SUMMARY`，便于回溯“为什么这次跑这些目标”。  
  - Starry 链路从 `make ARCH=<arch> run` 切换到 `scripts/ci-test.py <arch>`，降低长驻超时风险；增加 `disk.img` 检查与缺失兜底处理。  
  - 增补 `python3`、`cmake`、交叉编译器等前置依赖检查，并对 git 网络参数做抗抖动配置，提升 CI 稳定性。

### EEVDF 相关工作

- **EEVDF per-task 实现要点**  
  - 采用 per-task 状态建模，每个任务维护独立的 `vruntime`、`deadline` 与时间片信息，避免仅依赖全局近似导致的判定偏差。  
  - `V` 作为运行资格基准，采用就绪队列 `vruntime` 的加权平均动态更新；调度时先判断任务是否 eligible，再比较 `deadline`。  
  - 任务选择遵循“先满足 eligible，再比较 deadline”的策略；当就绪队列出现更应当运行的任务时触发 deadline 驱动抢占。
  - 支持运行中优先级（nice）调整，权重变化可及时反映到后续调度决策，保证策略一致性。  
  - 增加可开关统计能力，用于观测调度路径命中情况并辅助回归验证。
- **EEVDF 单元测试（`eevdf_tests`）**  
  - 在仓库根目录执行 `cargo test -p axsched eevdf_tests -- --nocapture`。  
  - `eevdf_tests` 共 **11 项测试通过**，覆盖高权重更早 deadline、eligible 约束优先、deadline 抢占、被抢占任务 deadline 保留、运行中优先级调整与非法 nice 拒绝等核心语义。  
  - 增补统计能力相关验证，包含 `picks_total`、`preempt_by_deadline`、`fallback_no_eligible`、`slice_expired` 等计数正确性。
- **统计可观测性增强**  
  - per-task EEVDF 统计支持开关与读取：`set_stats_enabled(bool)`、`stats()`、`reset_stats()`。  
  - 对低概率分支采用测试态覆盖，保证测试充分性且不影响正式构建行为。
- **nice 与前台延迟演示（4x yes 对照）**  
  - 在单核 QEMU 保持相同后台负载（4 个 `yes`），仅调整 nice，观测前台 `time ls`：  
    - 4x 默认 `yes`：约 `0.6s`  
    - 4x `nice -n 19 yes`：约 `0.05s`  
    - 4x `nice -n -10 yes`：约 `2~3s`
  - 结论：在相同负载下，后台 nice 越低（优先级越高），对前台交互挤压越明显，延迟显著上升。

## 实现总结

1. 围绕 **多组织共享测试环境**，形成了“文件锁串行 + watcher 取消态解锁 + compose 无感启停”的闭环，显著降低了 Cancel 导致的锁阻塞风险。
2. 围绕 **依赖感知测试目标选择**，完成了“规则外置 + Rust 引擎优先 + shell 回退”的可维护方案，并在 CI 中打通自动检测、可解释输出与稳定性加固。
3. 通过主线模块化成果合并，减少了分叉维护成本，使后续规则迭代和目标扩展可在统一框架内持续推进。
4. 补充 EEVDF 语义测试与整机演示材料，形成“单元测试 + 运行时观测”的双证据。
